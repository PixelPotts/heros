Current State Assessment

Functional (Real)

  • Window manager — drag, resize (8-direction), minimize, maximize, close, z-order
  • AppContent abstract interface — clean virtual dispatch for render + input
  • Event routing — mouse, keyboard, scroll, text input to focused window
  • Journal app — full text editor, word-wrap, cursor, CRUD, filesystem persistence
  • Finance app — dashboard layout with donut charts, progress bars, cards
  • Frosted glass compositing pipeline with 8x downscale blur
  • Font system with 6 size tiers
  • Drawing primitives library (31 icons, circles, rounded rects, glow, lines)
  • Hit-test system for window chrome (titlebar, buttons, resize edges, content)
  • Hardware-accelerated rendering with vsync

Missing / Hardcoded

  • No app registry — apps instantiated directly via make_unique
  • Dock is static 7-icon array — only Journal icon (index 3) is wired
  • Left sidebar nav items are decorative — no click handlers
  • Right sidebar widgets show static fake data
  • No process model — no PIDs, no process state machine, no task manager
  • No file system abstraction — Journal uses raw POSIX calls
  • No IPC, event bus, clipboard, or drag-and-drop between apps
  • No settings/preferences system
  • No notification system — bell icon is decorative
  • No theming — colors hardcoded in every render function
  • No widget library — each app draws everything from raw primitives
  • No keyboard shortcuts framework
  • No search/spotlight capability
  • Finance app has zero persistence — entirely hardcoded data

App Infrastructure
Transform hardcoded demo into extensible, registry-driven OS
CRITICAL • START HERE
1A App Manifest & Registry System
Define what an "app" is at the OS level. Every app must have a manifest that describes its identity, capabilities, and constraints. The registry is the single source of truth for all installed apps.
  • Define AppManifest struct: app_id, name, icon, category, versionUnique string identifier (e.g., "com.heros.journal"), human-readable name, Icon enum, category for launcher grouping
  • Add window hints to manifest: default_rect, min_size, max_size, default_flagsApps declare their preferred initial size and capabilities (resizable, closable, etc.)
  • Add instance policy: singleton vs multi_instance flagJournal = singleton (one instance), TextEditor = multi-instance (one per file)
  • Add launch hints: start_maximized, start_centered, dock_pinned, autostartControls how the app appears on first launch and whether it's pinned to dock by default
  • Create AppRegistry class with register_app(), get_manifest(), list_apps(), list_by_category()Central registry that owns all manifests. Static initialization pattern for built-in apps.
  • Add app categories enum: Productivity, System, Utility, Creative, CommunicationUsed by launcher and sidebar to group apps logically
  • Migrate Journal and Finance to use manifests instead of hardcoded setup_default_windows()Remove all hardcoded app instantiation from ui.cpp
  • Add manifest validation on registration — reject duplicates, enforce required fieldsFail fast on malformed manifests during initialization
1B App Factory & Instantiation
Decouple app creation from window creation. A single code path for launching any app, used by dock, sidebar, keyboard shortcuts, and any future launcher.
  • Define AppFactory function signature: std::function<unique_ptr<AppContent>()>Each app registers a factory lambda that constructs its AppContent subclass
  • Add register_factory(app_id, factory_fn) to AppRegistryAssociates a creation function with each manifest entry
  • Implement AppRegistry::create(app_id) → unique_ptr<AppContent>Looks up factory, calls it, returns new instance. Returns nullptr if unknown app_id.
  • Implement AppRegistry::launch(app_id) — high-level launch that checks singleton, creates window via WMIf singleton and already running, focus existing window instead of creating new one
  • Add launch arguments support: launch(app_id, LaunchArgs) for opening specific files/viewse.g., launch journal with a specific entry pre-selected, or finance with a specific tab
  • Wire AppContent back-pointer: each app gets a reference to its owning process/window contextApps need to know their own window ID to request close, title change, etc.
  • Add AppContent::set_context(AppContext*) called after constructionAppContext provides: window_id, app_id, reference to system services
  • Create register_builtin_apps() function that registers Journal, Finance, and future built-insCalled once at startup, replaces setup_default_windows
1C App Lifecycle Signals
Apps need to know when they're activated, deactivated, about to close, or resized. Currently the only signal is destruction via unique_ptr reset.
  • Add on_activate() / on_deactivate() to AppContentCalled when window gains/loses focus. Apps can pause animations, save draft state, etc.
  • Add on_close() → bool to AppContent — return false to cancel closeJournal can prompt "save changes?" before allowing close. Finance can warn about unsaved edits.
  • Add on_resize(int w, int h) to AppContentNotify app of content area resize so it can rebuild layout caches (e.g., Journal's word-wrap cache)
  • Add on_launch(const LaunchArgs&) to AppContentCalled once after construction with any launch arguments (file to open, view to show)
  • Add on_save_state() → json and on_restore_state(json)Serialize/deserialize app state for session restore. Journal saves open entry, cursor position, scroll.
  • Add on_timer(uint32_t timer_id) for periodic callbacksReplace raw SDL tick comparisons (cursor blink) with proper timer registration
  • Wire WindowManager to call lifecycle methods at correct momentsset_focus → on_activate/on_deactivate, close_window → on_close check, resize end → on_resize
  • Migrate Journal's destructor-based save to on_close() lifecycle hookProper save with error handling instead of relying on unique_ptr destructor side effects
1D Dynamic Dock & App Launcher
The dock becomes data-driven from the registry. Every registered app with dock_pinned=true appears. Running apps show indicators. Clicking launches or focuses.
  • Refactor dock rendering to read pinned apps from AppRegistryReplace static Icon[] array with dynamic list from registry where dock_pinned=true
  • Show running (non-pinned) apps in dock dynamicallyPinned apps always show; running non-pinned apps appear temporarily
  • Implement dock click handler for all apps — not just JournalClick: launch if not running, focus if running, minimize if focused. Same behavior for every app.
  • Add dock indicator states: running (dot), minimized (dim dot), focused (bright dot)Visual feedback for app state directly on dock icons
  • Add dock tooltips on hover — show app nameHover over dock icon shows app name from manifest
  • Wire left sidebar nav items to launch apps by category"Journal" in sidebar opens Journal app; "Work" opens Finance; etc.
  • Implement dock right-click context menu: "Close", "New Window" (if multi-instance), "Pin/Unpin"Requires context menu infrastructure (can be simple initially)
  • Add dock badge support — notification count overlay on app iconsSmall number badge (e.g., "3" on Journal for unread entries)
  • Add dock auto-sizing — expand width as more apps are pinned/runningCurrently fixed at 380px; should grow dynamically based on icon count

1E App Context & System Service Access
Apps need a clean interface to request OS services without tightly coupling to system internals. AppContext is the app's handle to the world.
  • Define AppContext struct: window_id, app_id, pointers to system servicesPassed to each app after creation. The app's only interface to OS-level functionality.
  • Add ctx.request_close() — app asks the OS to close its own windowGoes through proper lifecycle (on_close check) instead of apps closing themselves directly
  • Add ctx.set_title(string) — app can update its window title dynamicallyJournal changes title to "Journal - [entry name]" when editing an entry
  • Add ctx.request_focus() — app can request the OS to bring it to frontFor apps that receive background notifications and want user attention
  • Add ctx.launch_app(app_id, args) — one app can launch anotherJournal could launch a Settings page; Finance could launch a detailed report view
  • Add ctx.get_system_time(), ctx.get_screen_size() — system info queriesApps use this instead of calling SDL directly, maintaining abstraction boundary

Process Model
Give each running app identity, state, and managed lifecycle
HIGH PRIORITY
2A Process Table & PID Assignment
Every running app gets a unique process ID and tracked state. The ProcessManager is the authority on what's running.
  • Define ProcessInfo struct: pid, app_id, state, window_id, start_time, cpu_timeCore metadata for each running process. PID is monotonically increasing uint32.
  • Define process states: Starting → Running → Suspended → Terminating → DeadState machine with well-defined transitions. Invalid transitions are rejected.
  • Create ProcessManager class with spawn(), terminate(), kill(), suspend(), resume()Central process lifecycle management. spawn() assigns PID, creates AppContent, opens window.
  • Add list_processes(), get_process(pid), find_by_app_id() queriesUsed by task manager, dock, and system services to inspect running state
  • Route all app launches through ProcessManager instead of direct WindowManager callsProcessManager owns the lifecycle: spawn → create AppContent → open_window → wire context
  • Add process parent tracking: parent_pid field for apps launched by other appsEnables process trees — when parent closes, children can optionally close too
  • Add process resource tracking: memory estimate, render time per frameEven if approximate, gives the system monitor real data to display
  • Emit process events: ProcessSpawned, ProcessTerminated, ProcessStateChangedOther system components (dock, taskbar, task manager) react to these events
2B Graceful Shutdown & Error Recovery
Closing an app should be a negotiated process, not an abrupt destruction. Apps can save, prompt, or refuse.
  • Implement graceful terminate: terminate(pid) → on_close() → wait → destroySends close request to app, waits for response, then removes window and process
  • Implement force kill: kill(pid) → immediate destroy, no callbacksFor unresponsive apps. Bypasses on_close(), directly destroys. Last resort.
  • Add close timeout: if on_close() doesn't return within 3 seconds, offer force-killPrevents apps from blocking shutdown indefinitely
  • Implement system shutdown sequence: terminate all in reverse-launch orderWhen user quits HerOS, all apps get orderly close signals
  • Add crash detection: wrap app render/event calls in try/catch, mark process as crashedA crashing app shouldn't bring down the whole OS. Show error dialog, offer restart.
  • Add restart capability: restart(pid) — terminate then re-spawn same app with saved stateUseful for apps that crash or need settings reload
  • Add "unsaved changes" dialog infrastructureWhen on_close returns false, system shows a modal dialog: Save / Discard / Cancel
2C Background Services
Not all processes need windows. Services run in the background providing capabilities to the OS and other apps.
  • Define Service base class: on_start(), on_stop(), on_tick()Like AppContent but without render/input methods. Runs headless.
  • Add service lifecycle: Starting → Running → Stopped with restart policyServices can be configured to auto-restart on crash
  • Create ClockService — maintains system time, emits MinuteChanged, HourChanged eventsTopbar clock reads from this service instead of calling time() directly
  • Create NotificationService stub — queue and display toast notificationsApps call notify(), service manages queue, display duration, and dismissal
  • Create SystemMonitorService — tracks frame time, process count, memory estimatesFeeds data to the right sidebar "System Harmony" widget, making it real
  • Add service discovery: ProcessManager::get_service<T>() typed accessorApps and other services can locate services by type
  • Add service dependency declaration in manifest: requires: ["notification", "clock"]Services start in dependency order; dependent services start before dependents
2D Session Management
Remember what was running between sessions. When HerOS starts, restore the previous workspace.
  • Save session state on shutdown: list of running apps, window positions, statesWrite to ~/.heros/session.json: [{app_id, rect, maximized, z_order, app_state}, ...]
  • Restore session on startup: re-launch saved apps with saved positionsReplace setup_default_windows with session restore. Fall back to defaults if no session.
  • Per-app state serialization via on_save_state() / on_restore_state()Journal saves: open entry index, cursor position, scroll offset, active tab
  • Add "Default Session" concept: what apps open on first-ever launchDefined in code or config: e.g., Finance maximized + Journal minimized
  • Add session auto-save on a timer (every 60 seconds) for crash recoveryIf HerOS crashes, next launch restores from last auto-save

File & Data Layer
Persistence becomes an OS-provided service, not each app's problem
HIGH PRIORITY
3A Virtual Filesystem (VFS)
Abstract filesystem access behind a clean API. Apps never call POSIX directly. The VFS maps virtual paths to real disk locations.
  • Define FileSystem interface: read(), write(), list(), mkdir(), remove(), stat(), exists()Pure virtual interface. Default implementation wraps POSIX. Could be swapped for in-memory FS for testing.
  • Define virtual path layout: /system/, /apps/<id>/, /user/, /tmp//system/ = OS config. /apps/journal/data/ = Journal's data. /user/ = user files. /tmp/ = ephemeral.
  • Implement path resolution: virtual paths → real paths under ~/.heros//apps/journal/data/entry1.txt → ~/.heros/apps/journal/data/entry1.txt
  • Add ScopedFS: per-app filesystem view scoped to /apps/<id>/Apps get a ScopedFS that can only access their own data directory. Sandboxing.
  • Add FileInfo struct: path, size, modified_time, is_directory, mime_typeReturned by stat() and list(). MIME type inferred from extension.
  • Migrate Journal from raw POSIX to VFS APIReplace data_dir() + fopen/fclose with fs.read(), fs.write(), fs.list()
  • Add FileWatcher: notify apps when files in their data directory changeUses inotify on Linux. Enables live-reload of data when edited externally.
  • Add atomic write: write_atomic(path, data) — write to .tmp then renamePrevents data corruption if HerOS crashes mid-write
3B Settings & Preferences Store
Typed key-value store for system and per-app settings. JSON-backed, observable, with defaults.
  • Define Settings class: get<T>(key, default), set(key, value), has(key), remove(key)Typed accessors for bool, int, float, string, color. Returns default if key doesn't exist.
  • Add namespace scoping: settings.system(), settings.app("journal")System settings in /system/settings.json, per-app in /apps/journal/settings.json
  • Add change observers: settings.on_change(key, callback)Theme change immediately updates all windows without restart
  • Define system settings schema: wallpaper_path, theme, dock_position, font_scale, quiet_hoursWell-known keys that system components read from
  • Add settings migration: version field + upgrade functions for schema changesWhen settings format changes between versions, migrate automatically
  • Implement JSON serialization/deserialization for settings filesUse a lightweight JSON library or simple parser. Read on startup, write on change.
  • Add settings import/export for user portabilityExport all settings to a single JSON file, import to restore on new system
3C File Associations & MIME Types
Map file types to apps. The OS knows which app handles which files.
  • Define MIME type registry: extension → MIME type mapping.txt → text/plain, .journal → application/x-heros-journal, .csv → text/csv
  • Add handles field to AppManifest: list of MIME types the app can openJournal handles text/plain and application/x-heros-journal
  • Implement open_file(path) — resolve MIME type, find handler app, launch with file pathThe OS-level "double-click a file" operation
  • Add default handler per MIME type + user override capabilitySystem picks first registered handler; user can change default in settings
  • Add "Open With..." dialog for choosing non-default handlerRight-click file → Open With → shows all apps that handle that MIME type
  • Add AppContent::on_open_file(path) lifecycle methodCalled when an app is launched to handle a specific file
3D Data Persistence for Finance App
Make Finance real. Replace hardcoded data with persistent storage using the VFS and settings systems.
  • Define Finance data model: Account, Transaction, Budget, Goal structsProper data structures replacing inline hardcoded values
  • Implement Finance data persistence via VFS: /apps/finance/data/Accounts in accounts.json, transactions in transactions.json, etc.
